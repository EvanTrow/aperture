# Aperture v0.7.6 Release Notes

This release improves **deployment** (multi-arch Docker images for **amd64** and **arm64**), **user sync** with Emby/Jellyfin (respect **disabled** accounts), **setup wizard** job triggers, and **STRM library lifecycle** (automatic removal of virtual libraries and output folders when recommendations are turned off). **Baseline:** follows **v0.7.5**.

---

## At a glance

| Theme | What shipped |
| --- | --- |
| **Docker / CI** | GitHub Actions builds and pushes **linux/amd64** and **linux/arm64** images to GHCR; QEMU for cross-arch builds; root **`.dockerignore`** to shrink build context. |
| **Media server users** | New **`users.provider_disabled`** column, synced from Emby/Jellyfin **Policy.IsDisabled**; scheduled recommendation and watch-history jobs skip these users. |
| **Setup wizard** | OpenAPI route param for **POST `/api/setup/jobs/:name/run`** allows all jobs the handler already supported (e.g. watch history, embeddings), fixing **400** errors during initial jobs. |
| **STRM libraries** | When an admin disables recommendations (or a user becomes **provider-disabled** during `sync-users`), Aperture can **delete** the virtual library in Emby/Jellyfin, **remove** the user’s output directories under the Aperture libraries mount, and **clear** `strm_libraries` rows; library sync jobs **reconcile** stale rows. |

---

## Summary for users

- **Apple Silicon Macs** and **64-bit ARM** servers (e.g. Raspberry Pi with a 64-bit OS) can run the same **pre-built** container image as x86_64; Docker pulls the matching architecture automatically.
- If your server admin **disables** your Emby/Jellyfin account, Aperture stops treating you as eligible for recommendation jobs once **`sync-users`** runs (default every 30 minutes).
- **Initial setup** jobs such as **Sync Movie Watch History** no longer fail with a validation error when started from the wizard.

---

## Summary for admins

### Disabling recommendations and STRM output

When you turn off **movies**, **series**, or **all** recommendations for a user (**Admin → Users**), or when **`sync-users`** detects a user **disabled on the media server**, Aperture attempts to:

1. **DELETE** the corresponding **virtual library** in Emby/Jellyfin (by library name).
2. **Remove** that user’s folders under **`/aperture-libraries/aperture`** and **`/aperture-libraries/aperture-tv`** (inside the Aperture container).
3. **Delete** matching rows in **`strm_libraries`**.

Failures talking to the media server (e.g. library already removed) are logged; database and filesystem cleanup still proceed where possible. The scheduled **sync-movie-libraries** / **sync-series-libraries** jobs also run a **reconciliation** pass to clean up stale library records.

### Docker image architectures

Published images are **multi-architecture manifests**. No separate tag is required per CPU; use your existing **`ghcr.io/.../aperture`** image reference.

**Note:** **32-bit ARM** (`arm/v7`) is not a build target; use a **64-bit** OS on Raspberry Pi for parity with dependencies (e.g. native modules).

## API notes (operators)

- **Setup jobs:** `POST /api/setup/jobs/:name/run` — the documented allowed **`name`** values now match the server (includes watch-history, embedding, library, and related job ids).

---

## Update instructions

### For Docker users

```bash
docker compose pull
docker compose up -d
```

Apply database migrations per your process (`pnpm db:migrate` or automatic on startup, depending on deployment).

### Operator checklist

1. **Run migrations** — **`0113`** must apply before relying on **`provider_disabled`** behavior.
2. **Pull the new image** — multi-arch; no change to compose image name, but **`pull`** is required to get **arm64** if you are moving to ARM hardware.
3. **Expect one-time sync** — existing users get **`provider_disabled = false`** until **`sync-users`** runs and reads the current media-server flags.
